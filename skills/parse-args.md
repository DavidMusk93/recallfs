# 参数解析

## 目标

为脚本或命令行工具实现清晰、稳定、可维护的参数解析逻辑。优先使用 `getopt` / `getopts` / `getlongopt` 风格的显式解析方式，避免依赖隐式位置或难以扩展的手写分支。

## 适用场景

- 新增 shell、Python、Go、Node.js 等命令行入口。
- 为已有脚本补充 short option 和 long option。
- 需要统一默认值、必填参数、布尔开关、重复参数或透传参数。
- 需要让自动化 agent 能稳定调用脚本，避免交互式输入。

## 实现要求

- 同时考虑 short option 与 long option，例如 `-m` / `--model`。
- 参数命名要语义清晰，long option 使用完整含义，short option 保持简洁但不牺牲可读性。
- 默认值集中声明，避免散落在分支逻辑中。
- 参数解析与业务执行分离，建议拆成 `parse_args`、`validate_args`、`main` 等阶段。
- 必填参数缺失时给出明确错误，并输出 usage。
- 布尔参数使用正向语义，例如 `--skip-permissions`，避免双重否定。
- 对可能包含空格的参数保持引用安全，尤其是 shell 脚本中的 prompt、路径和额外参数。
- 支持 `--help` / `-h`，帮助内容应包含默认值、示例和退出码。
- 解析完成后只使用规范化后的变量，避免继续读取原始 `$@` 或未校验输入。

## Agent 友好约束

- 不要求用户手动输入；需要输入时优先支持环境变量、命令行参数或配置文件。
- 默认行为应可直接运行；危险操作必须显式配置或有清晰开关。
- 错误信息写入 stderr，正常结果写入 stdout，便于 agent 判断执行结果。
- 退出码必须稳定：成功为 `0`，参数错误建议为 `2`，运行时错误为非零。
- 示例命令必须可复制执行，不依赖未说明的上下文。

## 校验清单

- `--help` 输出准确且不执行主逻辑。
- short option 与 long option 行为一致。
- 默认值、环境变量和命令行参数的优先级明确。
- 未知参数会失败并提示具体参数名。
- 必填参数缺失会失败并提示缺失项。
- 带空格、特殊字符、空字符串的参数处理安全。
- 重复参数、透传参数或列表参数有明确合并策略。

## 输出建议

实现完成后，说明以下内容：

- 新增或修改了哪些参数。
- 每个参数的默认值和优先级。
- 典型调用示例。
- 已执行的校验命令。
