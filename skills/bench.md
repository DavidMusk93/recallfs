# bench

定义 bench，结果导向。比如：

- 吞吐量是否提升
- 耗时是否下降
- p99 / tail latency 是否改善
- CPU、内存、IO 是否更稳定

bench 流程

```text
+------------------------+
| 1. 明确目标与指标      |
| throughput / latency   |
| cpu / memory / p99     |
+-----------+------------+
            |
            v
+------------------------+
| 2. 固定 bench 环境     |
| 输入数据、机器、编译   |
| 参数、并发度保持一致   |
+-----------+------------+
            |
            v
+------------------------+
| 3. 建立 baseline       |
| 先测当前版本，记录     |
| 原始结果和波动范围     |
+-----------+------------+
            |
            v
+------------------------+
| 4. 理解代码与定位瓶颈  |
| trace / perf / profile |
+-----------+------------+
            |
            v
+------------------------+
| 5. 形成优化假设与 plan |
| 明确收益点与风险点     |
+-----------+------------+
            |
            v
+------------------------+
| 6. 小步实现            |
| 一次只改一个变量       |
+-----------+------------+
            |
            v
+------------------------+
| 7. 复测并对比 baseline |
| 看收益是否稳定可复现   |
+-----+-------------+----+
      |             |
   yes|             |no
      v             v
+-----------+   +----------------+
| 8. 提交    |   | 9. 回滚或重做  |
| 保留改动   |   | 调整假设与实现 |
+-----+-----+   +--------+-------+
      |                  |
      +--------+---------+
               |
               v
+------------------------+
| 10. 记录结果与经验     |
| 好坏结论都写入文档     |
| 保存命令、数据、图表   |
+------------------------+
```

补充原则

- 没有 baseline 的收益结论通常不可信。
- 没有固定环境的对比，容易把噪声当优化收益。
- 一次只改一个关键变量，否则无法判断收益来源。
- 收益不稳定时，不要急着提交，先检查样本量、波动和测量方法。
- 不仅记录成功经验，也记录失败尝试，避免重复走弯路。
