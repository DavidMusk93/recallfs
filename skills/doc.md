# doc

文档是 agent 的外部记忆。目标是沉淀上下文、保留证据、降低后续接手成本。

## 目录约定

```text
docs/
  analysis/   # 问题分析、根因定位、数据解释
  design/     # 方案设计、接口约定、架构取舍
  plan/       # 执行计划、迁移步骤、阶段拆分
  lesson/     # 复盘、踩坑、失败经验
  script/     # 临时脚本、常用命令、运行说明
  skill/      # 可复用工作方法和操作守则
```

命名建议：

- 使用小写英文和 `-`：`consumer-v2-backpressure.md`
- 标题直指主题，不用 `note.md`、`tmp.md`、`new.md`
- 同类文档放同目录，不按个人或日期随意散落

## 写作流程

```text
+--------------------------------------+
| 1. 明确文档目的                      |
| analysis / design / plan / lesson    |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 2. 选择目录和命名                    |
| 放到正确目录，文件名表达主题         |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 3. 写背景和结论                      |
| 先给读者结论，再展开证据             |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 4. 补充证据                          |
| 命令、数据、日志、现象要可复查       |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 5. 记录决策                          |
| 写清取舍、风险、未解决问题           |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 6. 追加更新                          |
| 新进展追加章节，不覆盖历史上下文     |
+--------------------------------------+
```

## 写入守则

- 统一使用 Markdown。
- 优先追加，不随意插入或覆盖历史记录。
- 多次更新时，用日期或阶段标题分隔。
- 分析结论必须有真实数据、日志或命令支撑。
- 使用 ASCII graph 表达流程。
- 能用表格表达对比关系，就不要用代码块。
- 代码块只放命令、配置、日志和必要片段。

## 分析文档

分析文档要回答三个问题：

- 现象是什么？
- 证据是什么？
- 结论和下一步是什么？

推荐结构：

```text
# title

## 结论
## 背景
## 现象
## 证据
## 分析
## 下一步
```

## 方案文档

方案文档要突出取舍，不要只堆实现细节。

推荐结构：

```text
# title

## 目标
## 非目标
## 当前问题
## 方案
## 流程
## 风险
## 验证
```

## 脚本文档

- `script/` 记录常用脚本、临时脚本和运行方式。
- Python 脚本必须说明 `venv` 使用方式。
- 脚本说明要包含输入、输出、依赖、执行命令。
- 危险脚本必须默认 `dry-run`，并说明 apply 模式。

## Agent 注意事项

- 先写结论，再写推导。
- 保留命令和原始数据，避免只写主观判断。
- 不确定的内容明确标注 `待确认`。
- 不要删除历史失败尝试，失败经验也有复用价值。
- 文档要服务后续 agent 接手，避免只写给当前上下文看。
