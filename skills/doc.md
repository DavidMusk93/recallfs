# doc

文档是 agent 的外部记忆。目标是沉淀上下文、保留证据、降低后续接手成本。

## 目录约定

```text
docs/
  analysis/   # 问题分析、根因定位、数据解释
  design/     # 方案设计、接口约定、架构取舍
  plan/       # 执行计划、迁移步骤、阶段拆分
  lesson/     # 复盘、踩坑、失败经验
  script/     # 临时脚本、常用命令、运行说明
  skill/      # 可复用工作方法和操作守则
```

命名建议：

- 使用小写英文和 `-`：`consumer-v2-backpressure.md`
- 标题直指主题，不用 `note.md`、`tmp.md`、`new.md`
- 同类文档放同目录，不按个人或日期随意散落

## 写作流程

```text
+--------------------------------------+
| 1. 明确文档目的                      |
| analysis / design / plan / lesson    |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 2. 选择目录和命名                    |
| 放到正确目录，文件名表达主题         |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 3. 写背景和结论                      |
| 先给读者结论，再展开证据             |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 4. 补充证据                          |
| 命令、数据、日志、现象要可复查       |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 5. 记录决策                          |
| 写清取舍、风险、未解决问题           |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 6. 追加更新                          |
| 新进展追加章节，不覆盖历史上下文     |
+--------------------------------------+
```

## 写入守则

- 统一使用 Markdown。
- 优先追加，不随意插入或覆盖历史记录。
- 多次更新时，用日期或阶段标题分隔。
- 分析结论必须有真实数据、日志或命令支撑。
- 使用 ASCII graph 表达流程。
- 能用表格表达对比关系，就不要用代码块。
- 代码块只放命令、配置、日志和必要片段。

## 章节编号

文档要有清晰章节分类和编号，让 Markdown 渲染后也容易快速定位。

- 一级标题只放文档主题。
- 二级标题使用编号，例如 `## 1. 结论`、`## 2. 背景`。
- 三级标题继承编号，例如 `### 2.1 现象`、`### 2.2 证据`。
- 同类内容放在同一章节，不要把背景、结论、证据混写。
- 短文档也至少保留 `结论`、`背景`、`下一步` 三类信息。

推荐编号结构：

```text
# 文档标题

## 1. 结论
## 2. 背景
## 3. 证据
### 3.1 命令
### 3.2 日志
## 4. 分析
## 5. 决策
## 6. 下一步
```

## ASCII Graph 规范

ASCII graph 要表达结构，不要只堆文字。实体必须使用 box wrap，层次和箭头要清晰。

- 实体使用 box 表达，例如 `client`、`server`、`worker`。
- 箭头表达方向，优先使用 `|`、`v`、`--->`。
- 主流程保持纵向向下，异常或分支使用 L 型侧向展开。
- box 宽度尽量统一，多行内容保持左对齐。
- 箭头旁标注动作或数据，例如 `/heartbeat(nodes[N])`。
- 不要让箭头穿过 box，也不要出现悬空线。

推荐写法：

```text
+-----------+
| client    |
+-----+-----+
      |
      | request
      v
+-----------+
| server    |
+-----+-----+
      |
      +---- fail ---->+-----------+
      |               | fallback  |
      |               +-----------+
      v
+-----------+
| success   |
+-----------+
```

避免写法：

```text
client -> server -> fallback/success
```

## 表格优先

对比、记录、参数、决策、风险优先使用表格，减少散乱段落。

适合表格的内容：

| 类型 | 推荐列 |
| --- | --- |
| 对比 | 方案、优点、缺点、结论 |
| 参数 | 参数、默认值、含义、影响 |
| 记录 | 时间、操作、结果、证据 |
| 风险 | 风险、触发条件、影响、应对 |
| 决策 | 决策项、选择、理由、后续动作 |

使用原则：

- 表格列数控制在 3 到 5 列。
- 单元格保持短句，长解释放到表格后。
- 表格用于结构化事实，不用于大段日志。
- 大段日志仍放代码块，并在表格中记录日志路径或关键摘要。

## 分析文档

分析文档要回答三个问题：

- 现象是什么？
- 证据是什么？
- 结论和下一步是什么？

推荐结构：

```text
# title

## 1. 结论
## 2. 背景
## 3. 现象
## 4. 证据
## 5. 分析
## 6. 下一步
```

## 方案文档

方案文档要突出取舍，不要只堆实现细节。

推荐结构：

```text
# title

## 1. 目标
## 2. 非目标
## 3. 当前问题
## 4. 方案
## 5. 流程
## 6. 风险
## 7. 验证
```

## 脚本文档

- `script/` 记录常用脚本、临时脚本和运行方式。
- Python 脚本必须说明 `venv` 使用方式。
- 脚本说明要包含输入、输出、依赖、执行命令。
- 危险脚本必须默认 `dry-run`，并说明 apply 模式。

## Agent 注意事项

- 先写结论，再写推导。
- 保留命令和原始数据，避免只写主观判断。
- 不确定的内容明确标注 `待确认`。
- 不要删除历史失败尝试，失败经验也有复用价值。
- 文档要服务后续 agent 接手，避免只写给当前上下文看。
- 写完后检查标题编号、ASCII graph 和表格是否符合规范。
