# Shell

目标：写高性能、稳健、可复现的 shell 命令和脚本，减少无谓 IO、避免脆弱管道、降低误操作风险。

## 1. 核心原则

- 先缩小输入范围，再处理内容。
- 优先使用能直接表达目标的命令，避免用通用文本替换硬凑。
- 避免读取大量文件；必须读取时先采样、过滤或限制行数。
- 所有路径、变量、参数都要加引号。
- 脚本入口使用 `set -euo pipefail`，并显式处理允许失败的命令。
- 参数解析、校验、执行逻辑分离，避免散乱分支。
- 危险操作默认 dry-run，apply 需要显式参数。

## 2. 命令选择

| 目标 | 推荐 | 避免 | 说明 |
| --- | --- | --- | --- |
| 查看 `/proc/<pid>/environ` | `strings` | `tr '\\0' '\\n'` | `strings` 更直接，避免手写替换 |
| 查找文件名 | `find` / `fd` | `ls \| grep` | 文件名可能包含空格和换行 |
| 搜索文本 | `rg` | `grep -R` | `rg` 默认更快，且有合理忽略规则 |
| 解析 JSON | `jq` | `grep` / `sed` | 结构化数据用结构化工具 |
| 解析 YAML | `yq` | `awk` 拼接 | 减少格式边界错误 |
| 查看大文件头尾 | `head` / `tail` | `cat` | 避免无意义全量输出 |
| 观察增长日志 | `tail -f` | 循环 `cat` | 降低 IO 和输出噪音 |
| 统计行数 | `wc -l < file` | `cat file \| wc -l` | 少一个进程和管道 |
| 批量处理文件 | `find -print0` + `xargs -0` | `for f in $(find ...)` | 安全处理特殊文件名 |

## 3. 性能流程

```text
+----------------------+
| 需要处理数据?         |
+----------+-----------+
           |
           v
+----------------------+
| 先限定目录/文件模式   |
+----------+-----------+
           |
           v
+----------------------+
| 先过滤再读取内容      |
+----------+-----------+
           |
           v
+----------------------+---- yes ---->+----------------------+
| 数据可能很大?         |              | head/tail/rg 限制范围 |
+----------+-----------+              +----------------------+
           | no
           v
+----------------------+
| 使用专用工具处理      |
+----------+-----------+
           |
           v
+----------------------+
| 输出摘要和证据路径    |
+----------------------+
```

## 4. 少读大文件

- 先用 `find`、`rg --files`、文件大小、修改时间缩小候选集。
- 对日志优先看 `tail -n 200`，不要直接 `cat`。
- 对未知大文件先用 `wc -c`、`file`、`head` 判断类型。
- 对压缩文件先列目录或采样，不要直接全量解压。
- 对仓库搜索优先加路径和 glob，例如 `rg "pattern" src/ -g '*.go'`。
- 输出给用户前做摘要，避免粘贴大量无效日志。

示例：

```bash
find logs -type f -name '*.log' -mtime -1 -size -100M
tail -n 200 logs/app.log
rg -n --max-count 20 'panic|error|failed' logs/
```

## 5. Robust Shell

脚本模板：

```bash
#!/usr/bin/env bash
set -euo pipefail

die() {
  echo "error: $*" >&2
  exit 2
}

main() {
  local input="${1:-}"
  [[ -n "$input" ]] || die "missing input"

  printf 'input=%s\n' "$input"
}

main "$@"
```

关键规则：

- 使用 `"$var"`，不要裸用 `$var`。
- 使用 `[[ ... ]]` 做条件判断。
- 使用 `local` 限定函数变量。
- 使用数组保存参数，避免字符串拼命令。
- 使用 `mktemp` 创建临时文件，不手写固定临时路径。
- 允许失败时显式写出，例如 `cmd || true`，并说明原因。
- 管道中需要读取失败时保留 `pipefail`，不要静默吞错。

## 6. 文件名安全

```bash
find "$root" -type f -name '*.log' -print0 |
  xargs -0 rg -n 'error|panic'
```

避免：

```bash
for f in $(find "$root" -type f); do
  grep error "$f"
done
```

原因：

- 文件名可能包含空格、换行、通配符。
- 命令替换会丢失边界。
- `find -print0` 和 `xargs -0` 能保留文件名边界。

## 7. 参数与交互

- agent 调用的脚本必须支持非交互参数。
- 必须支持 `--help` 或清晰 usage。
- 必填参数缺失时返回稳定错误码，建议 `2`。
- 不要在脚本中临时 `read -p` 等待用户输入。
- 需要确认时使用 `--apply`、`--yes`、`--dry-run` 这类显式参数。

## 8. 日志与输出

- 正常结果写 stdout。
- 错误和诊断写 stderr。
- 大任务输出日志文件，并在最终结果中给出路径。
- 输出要包含关键命令、输入范围、处理数量、失败数量。
- 不要把海量日志直接打到终端；先写文件，再摘要。

## 9. 危险操作

| 操作 | 要求 |
| --- | --- |
| 删除文件 | 默认 dry-run，明确列出目标 |
| 覆盖文件 | 先写临时文件，再原子替换 |
| 批量修改 | 先限制路径和后缀，再执行 |
| 远程同步 | 打印 source 和 destination |
| 后台任务 | 记录 PID、日志路径、停止方式 |

## 10. 检查清单

- 是否避免了不必要的全量读取？
- 是否对大文件使用了 `head`、`tail`、`rg` 或采样？
- 是否选择了专用工具，而不是脆弱文本替换？
- 是否所有变量和路径都正确引用？
- 是否处理了文件名空格和特殊字符？
- 是否支持非交互运行和 `--help`？
- 是否有稳定退出码和清晰错误信息？
- 是否记录了关键命令、输入范围和输出路径？

## 11. Agent 提醒

- shell 的第一目标是可靠完成任务，不是写最短命令。
- 性能问题通常先来自输入范围过大，再来自命令本身。
- 能不读就不读，能少读就少读，能用索引或过滤就不要全量扫描。
- 用正确工具表达正确语义，例如 environ 用 `strings`，JSON 用 `jq`。
