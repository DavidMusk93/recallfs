# preempt

目标：减少问答、确认和上下文切换，让 agent 长周期连续干活。

守则：

- 一次说清目标、边界、验收标准。
- 前置授权低风险操作，减少中途追问。
- 任务按阶段推进：探索、实现、验证、收尾。
- 调参时编译和试验解耦，避免反复改代码。

## 减少问答打断

- 开始前明确：改哪些文件、不改哪些文件、是否允许跑测试、是否允许联网、是否允许提交。
- 把“你自己决定即可”的范围一次说清。
- 长任务按阶段汇报，不要让 agent 每一步都停下来等指令。
- 成功标准要可判断，避免“优化一下”“整理一下”这种模糊目标。

## 避免手动确认

- 避免高破坏性操作，优先用版本管理、临时目录、重命名、移动代替删除。
- 脚本默认 `dry-run`，真正执行时再显式切换到 apply 模式。
- 命令尽量无交互化，比如加 `--yes`、`--non-interactive`、明确输入参数，不把 agent 卡在 prompt 上。
- 对低风险操作尽量做成默认直行，对高风险操作才保留确认。
- 批量操作前先收集一次确认，而不是每处理一个对象都弹一次确认。
- 把确认点放在关键边界，不要放在每一步。

## bench 设定目标

- bench 必须给出明确指标，不接受“优化一下”。
- 未达到目标前，agent 持续执行“定位 -> 修改 -> 验证”。
- 当前方向无收益时，及时收束并换方向。

目标示例：

- 吞吐提升 `20%`
- p99 降低 `15%`
- CPU 降到某个范围
- 内存峰值不超过某个阈值

### 补充 production-ready 检查项

- 确认无内存泄漏。
- 确认支持优雅退出。
- 确认自动扩缩容符合预期。
- 确认可观测体系已就位。

## 参数调优

原则：参数变化和代码变化分离；编译一次，批量试验。

```text
+--------------------------------------+
| 1. 明确目标指标                      |
| throughput / p99 / cpu / memory      |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 2. 找出关键参数                      |
| 只挑真正影响结果的 1~3 个参数        |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 3. 参数外提                          |
| 改成 env / flag / config             |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 4. 固定 bench 环境                   |
| 输入 / 机器 / 并发 / 编译选项固定    |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 5. 编译一次                          |
| 后续只改参数，不改代码               |
+--------------------------------------+
                   |
                   v
+--------------------------------------+
| 6. 批量 sweep 参数                   |
| 一次跑完多组组合                     |
+--------------------------------------+
                   |
                   v
+--------------------------------------+---- no ----->+----------------------------------+
| 7. 结果出现清晰趋势?                 |              | 扩大样本 / 缩小范围 / 重选参数  |
+--------------------------------------+              +----------------------------------+
                   | yes
                   v
+--------------------------------------+
| 8. 收敛到候选最优点                  |
+--------------------------------------+
                   |
                   v
+--------------------------------------+---- no ----->+----------------------------------+
| 9. 最优点收益稳定可复现?             |              | 回到 6 继续 sweep               |
+--------------------------------------+              +----------------------------------+
                   | yes
                   v
+--------------------------------------+
| 10. 固化默认值                       |
| 文档记录原因和结果                   |
+--------------------------------------+
```

- 一次只调少量关键参数，否则搜索空间爆炸，agent 很容易被拖进无穷试验。
- 优先把参数暴露成环境变量或配置，而不是硬编码在代码里。
- 能批量 sweep 就不要手工一次一次试。
- 记录每轮参数和结果，避免 agent 反复试过的组合。
- 先看趋势，再看最优值。没有趋势时，不要过早相信偶然的最好结果。
- 最终拿到最优配置后，再决定是否把它固化回代码默认值。

## 让 agent 持续工作的要点

- 任务描述完整。
- 低风险操作默认直行。
- bench 目标明确。
- 参数外提，编译和试验解耦。
- 长任务按阶段连续推进。

结论：preempt 的重点不是“恢复执行”，而是“减少被打断”。
