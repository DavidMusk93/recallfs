这题在现实里像什么 应用场景

抽象：在一堆「可索引的记录」里，找两条记录，使它们的某个数值指标满足二元关系（这里是求和）。

  • 订单列表里找两笔金额刚好凑满优惠门槛。
  • 选两门课，学分之和等于毕业要求的某一档。
  • 库存 SKU 两两配对：体积/重量之和接近装箱上限（同构于「配对」）。
  • 日志/事件流里找两个事件 id，使 latency 之和等于超时阈值。
  • 调试时：已知 sum，在数组快照里定位两个贡献者下标。
  • 编译器/分析：常量折叠前的「两操作数引用」粗匹配（教学类比）。

哈希不是为了「显得高级」。当你需要反复查询「有没有出现过某个值」， 且查询次数与 n 同阶时，用哈希是在付 O(n) 空间，换「少扫很多遍」的时间。 若 n 极小、或内存极紧、或不允许额外空间，暴力可能更合适——这是工程取舍，不是输赢。

多种解法 · 发散思维 对比

先理解每条路在优化什么，再选主推。主推 ≠ 唯一正确。

A. 暴力双循环
时间 O(n²) 空间 O(1) 易写易证

对每个 i，向后扫 j，检查和是否为 target。

何时够用：n 很小（如 < 100）、原型验证、面试先写对再优化。

思维：直接翻译题意，建立正确性基线。

B. 排序 + 双指针（变体）
时间 O(n log n) 下标需额外记录

若只关心「两个值」而非原下标，可排序后两头夹。本题要下标，需带着原 index 排序。

何时想：后续 3Sum/最接近三数等「有序空间上的配对」一族。

代价：排序改变了扫描顺序；实现比哈希啰嗦一点。

C. 一遍哈希（主推）
时间 O(n) 空间 O(n) 贴合题设

扫到 x 时查 need = target - x 是否已出现；先查后写。

在优化什么：把「在已见集合里找补数」从线性扫变成均摊 O(1) 查询。

应用迁移：任何「当前元素 + 历史集合」的配对问题。

D. 两遍哈希
时间 O(n) 空间 O(n) 同值需小心

先全部入库再查。同值两次（如 [3,3]）时，表要能区分两个下标或存列表。

教学价值：理解「索引结构先建后查」；实现上通常不如一遍干净。

解法换到了什么丢掉了什么
暴力零额外结构、思路直白时间
排序+双指针有序域上的线性扫原序；实现带索引时变重
哈希查询速度额外内存；哈希常数

发散：约束一变，解法怎么变 思维

  • 若要所有数对 → 不能早停；可能 O(n²) 输出规模。
  • 若数组有序且只要值 → 双指针更自然。
  • 若内存极紧 → 暴力或外部排序思路。
  • 若流式、只能看一遍且内存有限 → 需要近似/采样，不是原题哈希。

技巧服务于约束；先问「数据从哪来、有多大、要下标还是值、能否改数组」。

Q1. 本题要求返回的是？

解析：返回 indices；题设恰好一解。

Q2. nums=[3,2,4], target=6 的合法下标？（写成 i,j）

解析：2+4=6 → 下标 1 与 2。不是两个 3。

Q3. 一遍哈希主推解法时间复杂度？

解析：均摊 O(1) 查表 × n 次扫描。

Q4. 哪些说法正确？（多选）

解析：恰好一解；同一下标禁用；不同下标同值合法。

Q5. 扫到 x 时，在表里查的关键量常叫？（填 need / complement）

解析：补数 need = target − x，问「以前见过吗」。

Q6. 一遍哈希对当前 x 更稳妥的顺序？

解析：先查后写，避免把当前元素当自己的配对着。

Q7. 为什么本题值得学哈希，而不是「永远暴力就好」？

解析：正确性暴力也有；选哈希是因为约束下查询模式匹配，不是为了炫技。
