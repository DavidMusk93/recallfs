这题在现实里像什么 应用场景

抽象：在可索引记录里找 两条，使其数值满足二元关系（这里是求和）。

  • 订单列表里找两笔金额刚好凑满优惠门槛。
  • 选两门课，学分之和等于毕业要求的某一档。
  • 库存 SKU 两两配对：体积/重量之和接近装箱上限。
  • 日志流里找两个 event id，使 latency 之和等于超时阈值。
  • 调试：已知 sum，在数组快照里定位两个贡献者的 indices。
  • 分析：常量折叠前的「两操作数引用」粗匹配（教学类比）。

HashMap 不是为了「显得高级」。当需要反复查询「值是否出现过」， 且查询次数与 n 同阶时，用 O(n) 空间换查询时间。 n 极小或内存极紧时，brute force 可能更合适——工程取舍，不是输赢。

多种解法 · 发散思维 对比

先理解每条路在优化什么，再选主推。主推 ≠ 唯一正确。

A. Brute force 双循环
Time O(n²) Space O(1)

对每个 i，向后扫 j，检查和是否为 target。

何时够用：n 很小、原型验证、面试先写对再优化。

B. Sort + two-pointers（变体）
Time O(n log n) indices 需额外记录

若只要值不要 indices，可排序后两头夹。本题要 indices，需带着原 index 排序。

何时想：3Sum 等有序域配对一族。

C. One-pass HashMap（主推）
Time O(n) Space O(n)

扫到 x 时查 complement = target - x 是否已出现；先查后写。

在优化什么：「在 seen 集合找 complement」从线性扫变成均摊 O(1)。

D. Two-pass HashMap
Time O(n) Space O(n)

先全部入库再查。同值两次（如 [3,3]）时，表要能区分两个 indices。

解法换到了什么丢掉了什么
Brute force零额外结构、思路直白时间
Sort + two-pointers有序域上的线性扫原序；带 indices 时变重
HashMap查询速度额外内存；哈希常数

发散：约束一变，解法怎么变 思维

  • 若要所有数对 → 不能早停；可能 O(n²) 输出规模。
  • 若数组有序且只要值 → two-pointers 更自然。
  • 若内存极紧 → brute force 或外部排序思路。
  • 若流式、只能看一遍且内存有限 → 需要近似/采样，不是原题 HashMap。

技巧服务于约束；先问数据从哪来、有多大、要 indices 还是值。

Q1. 本题要求返回的是？

解析：返回 indices；题设恰好一解。

Q2. nums=[3,2,4], target=6 的合法 indices？（写成 i,j）

解析：2+4=6 → indices 1 与 2。不是两个 3。

Q3. One-pass HashMap 主推解法时间复杂度？

解析：均摊 O(1) 查表 × n 次扫描。

Q4. 哪些说法正确？（多选）

解析：恰好一解；同一 index 禁用；不同 index 同值合法。

Q5. 扫到 x 时，在 HashMap 里查的关键量英文名？（complement / need）

解析：complement = target − x，问「以前见过吗」。

Q6. One-pass 对当前 x 更稳妥的顺序？

解析：先查后写，避免把当前元素当自己的 pair。

Q7. 为什么本题值得学 HashMap，而不是「永远 brute force」？

解析：正确性暴力也有；选 HashMap 是因为查询模式匹配约束。
