# PID 2098966 写入进程 CPU 偏高分析

## 结论

PID `2098966` 是 `tide_worker` 写入进程，运行约 2h20m 后仍稳定消耗约 `350% CPU`，对应 3.5 个核左右。eBPF 采样显示，CPU 热点集中在 `fringedb::detail::BlockFilterIndex::Write` 的 hash block filter 索引转置写出逻辑，尤其是 `WriteHashBucketMajor` 内部的双层循环。

根因不是系统调用写入频率异常，也不是磁盘 IO 等待，而是 block filter index 在 flush 时采用 bucket-major 格式写出，需要按 `bucket_count = bitset_bytes * 8` 遍历每个 bucket，再遍历所有 row group，把 row-group-major bitset 转置为 bucket-major bitmap。高基数 hash block filter + 多 row group 场景下，这段逻辑 CPU 成本为 `O(bucket_count * rowgroup_count)`，会把写入 worker 打满。

关键源码：

- `src/fringedb/detail/block_filter_index.cc:62`：`WriteHashBucketMajor` 对每个 bucket 遍历所有 row group。
- `src/fringedb/detail/block_filter_index.cc:77`：采样落点，读取每个 row group 对应 bucket bit 位。
- `src/fringedb/detail/block_filter_index.cc:627`：`BlockFilterIndex::Write` 入口。
- `src/fringedb/detail/block_filter_index.cc:699`：写 hash rowgroup bitset 时调用 `WriteHashBucketMajor`。
- `src/fringedb/detail/block.cc:2532`：`IFlushFiles` 中 flush block filter index。
- `src/fringedb/detail/block.cc:3041`：`FlushBlockFilterIndex` 读取旧 summary、合并并重新写 swap 文件。
- `src/fringedb/detail/block.cc:3053`：调用 `BlockFilterIndex::Write(filelevel_swap_path, block_filter_index_->data())`。
- `src/fringedb/detail/block.cc:2705`：`FlushTimeout` 周期触发 flush。
- `src/fringedb/detail/block.cc:2721`：超时触发 `FlushFiles()`。

## 源码分析

flush 链路如下：

1. `WritableBlockImpl::Tick` 每秒定时运行。
2. `FlushTimeout` 判断有 row group 需要落盘，触发 `FlushFiles()`。
3. `IFlushFiles` 中发现 `block_filter_index_columns_` 不为空，调用 `FlushBlockFilterIndex()`。
4. `FlushBlockFilterIndex` 先读旧的 file-level summary 并 merge prefix，再调用 `BlockFilterIndex::Write` 重写 swap 文件。
5. `BlockFilterIndex::Write` 对 hash 类型 rowgroup bitset 调用 `WriteHashBucketMajor`。

其中 `WriteHashBucketMajor` 的核心逻辑是：

```cpp
const uint32_t bucket_count = static_cast<uint32_t>(bitset_bytes) * 8U;
const size_t rowgroup_bitmap_bytes = (rowgroup_bitsets.size() + 7U) / 8U;
for (uint32_t bucket = 0; bucket < bucket_count; ++bucket) {
  std::fill(bucket_bitmap.begin(), bucket_bitmap.end(), '\0');
  for (size_t rg = 0; rg < rowgroup_bitsets.size(); ++rg) {
    const auto& bitset = rowgroup_bitsets[rg];
    if (bucket < bitset.size() * 8U &&
        ((static_cast<uint8_t>(bitset[bucket >> 3U]) >> (bucket & 7U)) & 1U)) {
      bucket_bitmap[rg >> 3U] = ...;
    }
  }
  out.append(bucket_bitmap);
}
```

这里每次写 hash index 都会把所有 row group 的 bitset 做一次矩阵转置。若 bitset 较大、row group 数较多，即使写系统调用不多，也会产生大量用户态循环、分支和内存访问。eBPF 栈顶落在 `src/fringedb/detail/block_filter_index.cc:77`，正好对应读取每个 row group bitset 的内层循环。

## 为什么判断为 CPU-bound 而不是 IO-bound

- 进程状态以 running 为主，多个 writer 线程能吃满 CPU。
- bpftrace 用户态采样长期落在 C++ 用户态循环，而不是 `write`、`pwrite64`、`fsync`、内核 IO 路径。
- 写入增量约 `123.55 MiB/s`，但 `syscw` 只有约 `266/s`，系统调用频率不高。
- `BlockFilterIndex::Write` 单次耗时可达 `1~8s`，热点行是 bitset 转置读取，而不是 `ofstream::write`。

## 影响

- 写入 worker 被 `BlockFilterIndex::Write` 占用，导致写入进程整体 CPU 升高。
- `Tick -> FlushTimeout -> FlushFiles` 是周期路径，业务持续写入时会反复触发。
- 多个 table/block 同时 flush 时，多个 `*/w-*` worker 并行执行相同 CPU 重逻辑，表现为进程级多核打满。

## 建议

短期：

1. 对 CPU 异常实例确认是否开启了 hash block filter index，以及相关列的 bitset 大小、row group 数量。
2. 对高写入压力任务临时减少 block filter index hash 列数量，或降低触发频率，验证 CPU 是否回落。
3. 增加 `BlockFilterIndex::Write` 的耗时、输入规模指标，例如 `hash_count`、`rowgroup_count`、`bitset_bytes`、输出字节数。

中期优化方向：

1. 避免每次 flush 全量转置所有历史 row group：改为增量追加或分段持久化 bucket-major 数据。
2. 对稀疏 bitset 用 set-bit 迭代替代 `bucket_count * rowgroup_count` 全扫描。
3. 若必须保持当前文件格式，可按 column 并行转置，或用 SIMD/word-level 批处理降低内层循环成本。
4. `FlushBlockFilterIndex` 当前会读取旧 summary 并 merge prefix 后重写 swap 文件，建议评估是否可以只更新新增 row group，减少重复 CPU。
