# Tide Dev Agent 核心与可复用思维框架

## 结论

`tide-dev-agent` 的核心不是一个单纯的 CLI 工具，也不是一组脚本集合，而是一个面向软件工程 Agent 的“可验证交付操作系统”。

它把 Agent 做研发任务时最容易失控的部分拆成五层：

| 层次 | 作用 | 仓库中的主要承载 |
| --- | --- | --- |
| 协议层 | 规定 Agent 能做什么、不能做什么、按什么顺序做 | `AGENTS.md` |
| 技能层 | 把重复动作封装成稳定入口 | `skill/`, `.agents/skills/`, `.trae/skills/` |
| 项目执行层 | 对真实项目执行 clone/fetch/build/test/benchmark | `env/<project>/scripts/` |
| 证据门禁层 | 把“我做了”变成可复核 artifact | `ai/capabilities/`, `scripts/agent_capability_gate.py`, commit/MR gate |
| 回归演进层 | 把真实任务、失败和质量变化纳入长期反馈 | `ai/agent_regression/`, `ai/skill_regression/`, `ai/multi_agent/`, `memory/` |

可以把它理解成：

```text
+----------------------------------------------+
| 用户任务                                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 协议约束 AGENTS.md                          |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 角色分工 Commander/Designer/Coder/Evaluator |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 技能执行 skill + env scripts                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 真实仓库 build/test/benchmark               |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 证据 manifest / summary / logs / MR gate    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 回归 case / memory / pitfall / evolution    |
+----------------------------------------------+
```

这个项目最有价值的地方，是把 Agent 从“会说会写代码”推进到“能在工程约束下持续交付，并且交付过程可审计、可回归、可演进”。

## 核心是什么

### 1. Skill-First 的工程动作抽象

仓库把常见工程动作抽象为技能，而不是让 Agent 每次临场写命令。

典型动作包括：

| 动作 | 稳定入口 |
| --- | --- |
| 创建分支 | `skill/common/create_branch/` |
| 设计沉淀 | `skill/common/design/`, `skill/common/design_capture/` |
| 项目构建测试 | `skill/common/project/` + `env/<project>/scripts/` |
| 代码审查 | `skill/common/code_review/` |
| 提交、推送、MR | `skill/common/commit/`, `push/`, `open_mr/` |
| case 和 pitfall 沉淀 | `case_capture/`, `pitfall_capture/` |

这里的关键思想是：Agent 的自由度应该放在“理解、取舍、诊断、设计”上，而不是放在“每次猜 build/test 命令”上。

### 2. 真实项目验证，而不是模拟验证

`env/<project>/scripts/` 是项目执行层的核心。每个项目通过统一目录暴露标准动作：

```text
env/<project>/scripts/
  clone.sh
  fetch_dep.sh
  build.sh
  test.sh
  benchmark.sh
```

这带来几个重要收益：

| 问题 | 解决方式 |
| --- | --- |
| Agent 容易手写错误命令 | 所有项目动作收敛到标准脚本 |
| 不同项目工具链差异大 | 差异封装在 `env/<project>/scripts/` 内 |
| 验证结果不可复核 | 构建和测试必须走真实仓库、真实脚本 |
| Agent 可能用 mock 冒充通过 | 协议明确禁止伪仓、伪产物、伪验证 |

这个项目的工程信念很明确：研发 Agent 的可信度来自真实验证，不来自语言表达。

### 3. 多 Agent 不是“多几个模型”，而是职责隔离

`ai/multi_agent/` 把任务拆成多个角色：

| 角色 | 职责 |
| --- | --- |
| Commander | 判断可执行性、拆任务、监督流程、决定是否重试 |
| Designer | 设计方案、验收标准、风险和测试策略 |
| Coder | 实现、构建、测试、提交、MR |
| Evaluator | 判断结果是否值得保留 |
| Evolver | 分析低质量原因并生成下一轮改进指令 |

真正有意思的点不是“并行多人干活”，而是通过 `COCO_SKILL_DIR` 做角色技能隔离。每个角色只能看到自己该用的技能，降低职责漂移。

可复用抽象：

```text
+----------------------------------------------+
| 复杂任务                                     |
| 决策权 + 设计权 + 执行权 + 评价权 + 演进权  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 全部塞给一个 Agent                          |
| 容易自证正确、跳步骤、遗漏验证              |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 拆成多个角色                                |
| 明确输入、输出、能力边界和质量门槛          |
+----------------------------------------------+
```

### 4. SSoT：把模糊需求变成 0/1 验收

`docs/AI_Collaboration_Architecture_V2.md` 的核心是 SSoT，即 Single Source of Truth。它要求把需求、设计、验收标准转成明确施工图。

最重要的思想是：不能被 0/1 判定的需求，不应该直接进入执行。

例子：

| 模糊表达 | 可执行 SSoT 表达 |
| --- | --- |
| 性能更好 | 在指定 benchmark 上 TPS 提升不低于 X%，附 baseline 和实验结果 |
| 修复 crash | 给出复现命令、失败栈、修复后同命令退出码为 0 |
| 支持某功能 | 指定输入、输出、边界条件和负例测试 |
| 代码质量提升 | 指定 lint/test/code review gate 通过 |

这套方法适合迁移到任何复杂研发任务：先定义验收物理规则，再让 Agent 实现。

### 5. 能力目录：把协议变成机器可测能力

`ai/capabilities/core.json` 是很关键的设计。它把抽象能力变成机器可读目录：

| 能力 ID | 含义 |
| --- | --- |
| `CAP-SKILL-DISCOVERY` | skill 是否可发现、链接是否同步 |
| `CAP-MULTI-AGENT-ROLE-ISOLATION` | 多 Agent 角色技能是否隔离 |
| `CAP-PROJECT-BUILD-TEST` | 项目构建和测试是否通过统一入口真实执行 |
| `CAP-AGENT-REGRESSION-QUALITY` | 语言和质量回归是否能长期运行 |
| `CAP-COMMIT-GATE` | commit/MR 前是否通过静态和证据门禁 |

这代表一种非常重要的工程方法：

```text
+----------------------------------------------+
| 不要只写规则                                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 把规则映射到可检查对象                      |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 触发路径 / 必需 artifact / 自动化测试       |
| Jenkins job / pass-fail 判定                |
+----------------------------------------------+
```

只有这样，协议才不会停留在文档里。

### 6. 回归体系：评估 Agent，而不是相信 Agent

仓库把回归拆成两类：

| 回归类型 | 关注点 | 入口 |
| --- | --- | --- |
| 语言类回归 | 是否遵守协议、是否有边界意识、设计质量如何 | `ai/agent_regression/` |
| 技能类回归 | 技能背后的脚本是否真能跑通 | `ai/skill_regression/` |

`QUALITY_FRAMEWORK.md` 还进一步把质量分成维度：

| 维度 | 含义 |
| --- | --- |
| `workflow_compliance` | 是否遵守开发流程 |
| `context_safety` | 缺上下文时是否明确不能臆测 |
| `deliverable_completeness` | 是否给出路径、命令、产物、测试 |
| `validation_quality` | 验证矩阵是否充分 |
| `observability_and_risk` | 指标、风险、回滚是否明确 |
| `memory_capture_quality` | 是否沉淀 design/pitfall/memory |

这里的核心不是打分本身，而是把 Agent 能力变化变成趋势，避免“今天能做，明天退化”。

## coco 工具链重点

`coco` 是这个仓库里真正把“协议、技能、memory、项目脚本、回归任务”串起来的运行时入口。可以把 `scripts/coco_start.sh` 理解成 `coco` 的启动包装器和上下文注入器。

### 1. coco 启动总流程

```text
+-----------------------------+
| 用户执行 ./login.sh 或脚本   |
+-----------------------------+
              |
              v
+-----------------------------+
| 进入 dev container / workspace |
+-----------------------------+
              |
              v
+-----------------------------+
| scripts/coco_start.sh        |
+-----------------------------+
              |
              v
+-----------------------------+
| 检查 coco 命令和模型名       |
| coco models                  |
+-----------------------------+
              |
              v
+-----------------------------+
| 清理失效 local plugin        |
| coco plugin list/uninstall   |
+-----------------------------+
              |
              v
+-----------------------------+
| 安装并启用技能插件           |
| coco plugin install/enable   |
+-----------------------------+
              |
              v
+-----------------------------+
| 注入 Universal Agent Protocol |
| 可选注入 project memory       |
+-----------------------------+
              |
              v
+-----------------------------+
| 启动 coco                    |
| 交互模式或 -p 一次性任务      |
+-----------------------------+
```

### 2. coco 在本仓库中的几个入口

| 入口 | 作用 | 典型场景 |
| --- | --- | --- |
| `./login.sh` | 进入容器并默认启动 coco | 日常交互开发 |
| `scripts/coco_start.sh` | 安装技能插件、注入协议、启动 coco | 本仓库标准 coco bootstrap |
| `bash scripts/run_multi_agent_task.sh -P "..."` | 用 coco 拉起多 Agent 编排 | 复杂任务自动多轮执行 |
| `bash scripts/run_agent_regression.sh` | 用 coco 跑语言/协议回归 | daily/weekly Agent 能力检查 |
| `skill/common/project/scripts/run.sh` | 兼容性 project skill 分发器 | 从技能语义路由到 `env` 脚本 |

### 3. coco_start.sh 的核心机制

```text
+----------------------------------------------+
| scripts/coco_start.sh                       |
+----------------------------------------------+
                       |
      +----------------+----------------+
      |                |                |
      v                v                v
+-------------+  +-------------+  +-------------+
| 模型解析    |  | 技能目录解析 |  | 插件安装    |
| coco models |  | COCO_SKILL_DIR| | skill/.trae |
+-------------+  +-------------+  +-------------+
      |                |                |
      +----------------+----------------+
                       |
                       v
+----------------------------------------------+
| Memory bootstrap                            |
| constraints / README / agent_context / issue|
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| Protocol injection                          |
| Universal Agent Protocol V2 + CoC           |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| coco 执行                                   |
| 交互模式 / --json -p / run.log failure gate |
+----------------------------------------------+
```

### 4. coco 与技能插件的关系

`coco` 通过 plugin 机制暴露技能。`coco_start.sh` 会把本仓库的技能目录安装成本地插件。

```text
+----------------------------------------------+
| skill/                                       |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| coco plugin install --type local skill/     |
| --name tide-skills                          |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| coco plugin enable tide-skills              |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| Agent 在对话中可发现并调用技能              |
+----------------------------------------------+
```

工作区技能和项目技能也会被额外安装：

```text
+----------------------------------------------+
| .trae/skills/                                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| tide-skills-workspace-local                 |
+----------------------------------------------+

+----------------------------------------------+
| projects/<project>/.trae/skills/            |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| tide-skills-project-<project>               |
+----------------------------------------------+

+----------------------------------------------+
| external/skills/skills/                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| tide-skills-external                        |
+----------------------------------------------+
```

这里的设计点是：技能不是静态写死在模型里，而是在每次启动时由仓库状态动态安装到 coco 运行时。

### 5. COCO_SKILL_DIR 与多 Agent 角色隔离

多 Agent 编排里最值得关注的是 `COCO_SKILL_DIR`。`ai/multi_agent/orchestrator.py` 会为每个角色构造一个裁剪后的技能 bundle，然后用环境变量传给 `coco_start.sh`。

```text
+----------------------------------------------+
| role spec                                    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| allowed_skills: [...]                        |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| build_role_skill_bundle()                   |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| run_dir/round_XX/<role>/skills/             |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| COCO_SKILL_DIR=<that skill bundle>          |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| scripts/coco_start.sh                       |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| coco 只加载该角色允许使用的技能             |
+----------------------------------------------+
```

这让角色隔离从“prompt 约定”变成“运行时能力隔离”：

| 角色 | 关注点 | 技能可见性原则 |
| --- | --- | --- |
| Commander | 可执行性、拆解、重试 | 不给直接深度实现能力 |
| Designer | 设计、验收、风险、测试策略 | 给设计和任务拆解能力 |
| Coder | 实现、构建、测试、提交、MR | 给 develop/project/commit/push/open_mr |
| Evaluator | 判断是否保留 | 给 case/memory 类能力 |
| Evolver | 失败复盘、下轮改进 | 给 evolution/case/task 类能力 |

### 6. coco 与 project skill 的执行链路

`project` 技能是从 Agent 语义到真实项目脚本的桥。

```text
+----------------------------------------------+
| Agent 想执行 build/test                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| project skill                                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| skill/common/project/scripts/run.sh          |
+----------------------------------------------+
                       |
      +----------------+----------------+
      |                                 |
      v                                 v
+------------------+          +------------------+
| 解析 --action    |          | 解析 --project   |
+------------------+          | 显式参数/remote  |
                              +------------------+
                       |
                       v
+----------------------------------------------+
| 查找脚本                                     |
| 1. env/<project>/scripts/<action>.sh         |
| 2. projects/<project>/project/scripts/...    |
| 3. projects/<project>/scripts/<action>.sh    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| bash <script> <args>                         |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 真实 clone / fetch_dep / build / test        |
+----------------------------------------------+
```

因此，`coco` 不是直接等于“执行 shell 命令”。更准确地说：

```text
+----------------------------------------------+
| coco 对话意图                                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 技能选择                                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 统一 project 语义                            |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| env 标准脚本                                 |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 真实项目验证                                 |
+----------------------------------------------+
```

### 7. coco 一次性任务和交互任务

`coco_start.sh` 支持两种模式：

```text
+----------------------------------------------+
| 无 prompt                                    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| exec coco                                    |
| -c permission_mode=bypass_permissions        |
| -c "model.name=..."                         |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 进入交互模式                                 |
+----------------------------------------------+
```

```text
+----------------------------------------------+
| 有 prompt                                    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 注入 Universal Agent Protocol V2             |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| coco --json -p "<prompt>"                    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 输出 JSON stream                             |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 检查 coco_result/run.log                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 如果技能写入 failure，则强制 exit 1          |
+----------------------------------------------+
```

这对 Jenkins 很重要：不能只相信 `coco` 进程退出码，还要看技能运行过程中写入的失败状态。

### 8. 使用 coco 时最该关注的变量

| 变量 | 含义 | 典型用途 |
| --- | --- | --- |
| `COCO_SKILL_DIR` | 当前要加载的技能目录 | 多 Agent 角色隔离、测试裁剪技能 |
| `COCO_PLUGIN_NAME` | 本地技能插件名 | 避免不同任务插件名冲突 |
| `COCO_MODEL_NAME` | 模型名 | 指定模型或兼容模型名称修正 |
| `COCO_DISABLE_EXTERNAL_SKILLS` | 是否禁用外部技能 | 回归测试中减少不确定性 |
| `COCO_ENABLE_MEMORY_BOOTSTRAP` | 是否注入 memory | 需要项目上下文时打开 |
| `COCO_MEMORY_PROJECT` | 指定 memory 项目名 | 不在 `projects/<project>` 目录时手动指定 |
| `COCO_OUTPUT_MODE` | 输出模式 | Jenkins 或 final answer 模式 |

```text
+----------------------------------------------+
| coco 不是单个 CLI                            |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| coco 是 Agent runtime                        |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| plugin system: 技能发现                      |

| env vars: 运行时隔离                         |
| project skill: 真实工程动作                  |

+----------------------------------------------+
```

## 以 coco 为中心的服务端监听 Agent 方案

目标：部署一套长期运行的服务端 Agent，让外部系统可以提交任务，服务端监听任务、选择技能、启动隔离的 `coco` worker 执行，并把结果、日志、证据和经验沉淀回仓库或外部存储。

### 1. 总体架构

```text
+-----------------------------+
| 外部触发源                   |
| HTTP / Webhook / IM / Cron   |
+-----------------------------+
              |
              v
+-----------------------------+
| Agent Listener API           |
| 鉴权 / 限流 / 入队 / 查询    |
+-----------------------------+
              |
              v
+-----------------------------+
| Task Store                   |
| pending/running/done/failed  |
+-----------------------------+
              |
              v
+-----------------------------+
| Scheduler                    |
| 路由项目 / 选择模型 / 配技能 |
+-----------------------------+
              |
              v
+-----------------------------+
| Skill Control Plane          |
| registry / bundle / policy   |
+-----------------------------+
              |
              v
+-----------------------------+
| coco Worker Pool             |
| 每个任务一个隔离运行目录      |
+-----------------------------+
              |
              v
+-----------------------------+
| Evidence Store               |
| stdout/stderr/run.log/summary|
+-----------------------------+
              |
              v
+-----------------------------+
| Result Notifier              |
| Webhook / IM / MR comment    |
+-----------------------------+
```

这个架构里，`coco` 不直接暴露给用户。用户只提交“任务意图”，服务端负责把意图转成带协议、技能、memory、运行目录和证据回收的 `coco` 执行。

### 2. 核心组件职责

| 组件 | 职责 | MVP 形态 |
| --- | --- | --- |
| Listener API | 接收任务、查询状态、取消任务 | Python FastAPI 或 Go HTTP server |
| Task Store | 保存任务状态、参数、结果摘要 | SQLite 起步，后续 PostgreSQL |
| Scheduler | 决定由哪个 worker 执行、加载哪些技能 | 单进程队列起步 |
| Skill Control Plane | 管理技能来源、版本、启用策略、隔离 bundle | 基于 `skill/` + registry YAML |
| coco Worker | 调用 `scripts/coco_start.sh -p` 执行任务 | 子进程 worker |
| Workspace Manager | 准备真实 repo/worktree、分支、缓存 | 先复用当前 workspace，后续隔离 worktree |
| Evidence Store | 保存 JSON stream、run.log、summary、产物路径 | 本地 `runs/<task_id>/` |
| Notifier | 回传完成、失败、阻塞原因 | 先 stdout/log，后续 IM/webhook |

### 3. 服务端任务生命周期

```text
+-----------------------------+
| 1. submit task               |
| POST /tasks                  |
+-----------------------------+
              |
              v
+-----------------------------+
| 2. validate request          |
| 鉴权 / 参数 / project / risk |
+-----------------------------+
              |
              v
+-----------------------------+
| 3. create task record        |
| status=pending               |
+-----------------------------+
              |
              v
+-----------------------------+
| 4. resolve skill profile     |
| project + intent -> skills   |
+-----------------------------+
              |
              v
+-----------------------------+
| 5. prepare run directory     |
| runs/<task_id>/              |
+-----------------------------+
              |
              v
+-----------------------------+
| 6. prepare COCO env          |
| COCO_SKILL_DIR / MODEL / ... |
+-----------------------------+
              |
              v
+-----------------------------+
| 7. launch coco worker        |
| coco_start.sh -p prompt      |
+-----------------------------+
              |
              v
+-----------------------------+
| 8. stream logs and heartbeat |
| stdout/stderr/run.log        |
+-----------------------------+
              |
              v
+-----------------------------+
| 9. gate result               |
| exit code + evidence checks  |
+-----------------------------+
              |
              v
+-----------------------------+
| 10. persist and notify       |
| result / summary / artifacts |
+-----------------------------+
```

MVP 不要一开始做复杂平台，先保证“能稳定监听、能隔离技能、能保存证据、能取消任务”。

```text
+-----------------------------+
| Phase 0: 本地单机 MVP        |
+-----------------------------+
              |
              v
+-----------------------------+
| 1. 新增 server/agent_server.py |
+-----------------------------+
              |

+-----------------------------+

+-----------------------------+

+-----------------------------+

+-----------------------------+
              |

+-----------------------------+
```

### 5. 技能管理总原则

服务端监听 Agent 最容易失控的地方就是技能管理。核心原则是：技能必须有来源、有版本、有权限、有测试、有同步、有审计。

```text
+----------------------------------------------+
| skill source                                 |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| skill registry                               |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| policy check                                 |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| task skill profile                           |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| per-task skill bundle                        |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| COCO_SKILL_DIR                               |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| coco plugin install/enable                   |
+----------------------------------------------+
```

不要让服务端 worker 直接加载“所有技能”。每个任务都应该得到一个最小技能集合。

### 6. 技能来源分层

| 来源 | 路径 | 用途 | 是否默认加载 |
| --- | --- | --- | --- |
| 核心技能 | `skill/common/*` | develop/project/commit/push/MR/design 等稳定能力 | 按 profile 加载 |
| 工作区技能 | `skill/workspace/*` | bytedcli/jenkins/reset/高级设计等环境能力 | 默认不全量加载 |
| Trae 可见链接 | `.agents/skills/`, `.trae/skills/` | IDE/Trae/coco 可发现入口 | 由同步脚本生成 |
| 项目本地技能 | `projects/<project>/.trae/skills` | 项目私有能力 | 只对对应项目加载 |
| 外部技能 | `external/skills/skills` | 第三方或外部扩展 | 默认关闭，白名单开启 |

### 7. 技能 registry 设计

建议新增一个机器可读 registry，例如：

```text
server/skill_registry.yaml
```

示例结构：

```yaml
version: 1
skills:
  project:
    source: skill/common/project
    category: execution
    trust_level: core
    allowed_profiles: [analysis, develop, project_ops]
    requires_tests:
      - tests/test_project_skill_regression.py
    risk:
      filesystem_write: true
      network: false
      git_write: false

  commit:
    source: skill/common/commit
    category: delivery
    trust_level: core
    allowed_profiles: [delivery]
    requires_approval: true
    risk:
      filesystem_write: true
      git_write: true

profiles:
  analysis:
    include:
      - design
      - design_capture
      - project
    exclude:
      - commit
      - push
      - open_mr

  develop:
    include:
      - design
      - develop
      - project
      - code_review
    exclude:
      - push
      - open_mr

  delivery:
    include:
      - design
      - develop
      - project
      - code_review
      - commit
      - push
      - open_mr
```

这个 registry 的价值是让技能管理从“目录里有什么就能用什么”，变成“任务 profile 明确允许什么才能用什么”。

### 8. 每任务技能 bundle 生成

服务端不要直接设置 `COCO_SKILL_DIR=skill`。更安全的做法是为每个 task 生成临时 bundle。

```text
+----------------------------------------------+
| task_id                                      |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| skill_profile                                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| registry resolve                             |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| runs/<task_id>/skills/                       |
| design -> skill/common/design                |
| project -> skill/common/project              |
| code_review -> skill/common/code_review      |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| COCO_SKILL_DIR=runs/<task_id>/skills         |
+----------------------------------------------+
```

生成规则：

| 规则 | 说明 |
| --- | --- |
| 只链接 `SKILL.md` 所在目录 | 避免半成品技能进入运行时 |
| 禁止重名技能 | 当前 `.agents/.trae` 扁平化也要求无重名 |
| 默认禁用危险技能 | `commit/push/open_mr/bytedcli/jenkins` 需要 profile 或审批 |
| bundle 只读化 | 生成后 worker 不应修改技能定义 |
| 每个 bundle 写 manifest | 记录技能名、源路径、git commit、hash |

建议 bundle manifest：

```json
{
  "task_id": "task_20260527_001",
  "profile": "analysis",
  "repo_commit": "abcdef",
  "skills": [
    {
      "name": "project",
      "source": "skill/common/project",
      "sha256": "..."
    }
  ]
}
```

### 9. 技能同步与发布流程

当前仓库已有 `scripts/sync_skill_links.py`，它会从 `skill/**/SKILL.md` 同步到：

```text
.agents/skills/
.trae/skills/
```

服务端应该把它纳入技能发布流程：

```text
+----------------------------------------------+
| 修改 skill/                                  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| python3 scripts/sync_skill_links.py          |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| python3 -m unittest tests/test_skill_sync.py |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| python3 -m unittest test_workspace_registry  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 更新 skill_registry.yaml                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| server /skills/reload                        |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 新任务使用新技能 bundle                      |
+----------------------------------------------+
```

发布门禁：

| 门禁 | 说明 |
| --- | --- |
| link sync | `.agents/skills/` 和 `.trae/skills/` 必须同步 |
| duplicate check | 扁平化技能名不能重复 |
| tests required | skill 改动必须带 `tests/` |
| registry required | 新技能必须登记 category/profile/risk |
| dry-run task | 至少跑一个不写仓库的 analysis profile 任务 |

### 10. 技能权限模型

建议按风险分级，而不是只按技能名管理。

| 等级 | 能力 | 示例技能 | 默认策略 |
| --- | --- | --- | --- |
| R0 只读分析 | 读文件、总结、设计 | `design`, `design_capture` | 默认允许 |
| R1 本地执行 | build/test/benchmark | `project` | 允许但要记录命令 |
| R2 写文件 | 修改代码、写 memory | `develop`, `pitfall_capture` | 需要任务 profile |
| R3 Git 写入 | commit/push/MR | `commit`, `push`, `open_mr` | 默认禁止，需审批 |
| R4 外部系统 | Jenkins/Feishu/Bitable | `jenkins-ops`, `bytedcli` | 默认禁止，需白名单和凭据隔离 |
| R5 workspace 管理 | reset/reclone | `reset-workspace` | 仅管理员触发 |

服务端调度时的判断：

```text
+----------------------------------------------+
| task profile                                 |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| requested skills                             |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| risk level check                             |
+----------------------------------------------+
       |             |             |           |
       v             v             v           v
+------------+ +------------+ +------------+ +------------+
| R0/R1      | | R2         | | R3         | | R4/R5      |
| allow      | | trusted    | | approval   | | secret/admin|
+------------+ +------------+ +------------+ +------------+
```

### 11. 技能热更新策略

服务端监听 Agent 不建议让运行中的任务热更新技能。推荐策略：

```text
+----------------------------------------------+
| server start                                 |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| load registry snapshot                       |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| task starts                                  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| create immutable skill bundle                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| task uses that bundle until finish           |
+----------------------------------------------+
```

当技能发生更新：

```text
+----------------------------------------------+
| git pull / skill update                      |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| sync_skill_links.py                          |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| skill tests                                  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| /skills/reload                               |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| only new tasks use new registry snapshot     |
+----------------------------------------------+
```

这样可以避免一个任务执行到一半时技能定义变化，导致结果不可复现。

### 12. 服务端运行目录设计

建议每个任务都有独立运行目录：

```text
runs/
  task_20260527_001/
    request.json
    status.json
    prompt.txt
    env.json
    skills/
    skill_manifest.json
    stdout.log
    stderr.log
    coco_raw.jsonl
    coco_result/
      run.log
    evidence.json
    summary.md
```

关键文件语义：

| 文件 | 作用 |
| --- | --- |
| `request.json` | 原始请求，便于复现 |
| `prompt.txt` | 注入协议后的最终 prompt |
| `env.json` | `COCO_*` 等环境变量快照 |
| `skill_manifest.json` | 本任务实际加载了哪些技能 |
| `coco_raw.jsonl` | coco 原始 JSON stream |
| `coco_result/run.log` | 技能写入的运行状态 |
| `evidence.json` | 退出码、日志路径、测试数量、产物 |
| `summary.md` | 给人看的结果摘要 |

### 13. Worker 执行命令模板

服务端 worker 最终可以落到类似命令：

```bash
COCO_SKILL_DIR="/path/to/runs/task_001/skills" \
COCO_PLUGIN_NAME="tide-skills-task-001" \
COCO_MODEL_NAME="GPT-5.4" \
COCO_DISABLE_EXTERNAL_SKILLS=1 \
COCO_ENABLE_MEMORY_BOOTSTRAP=1 \
COCO_MEMORY_PROJECT="tide-dev-agent" \
COCO_OUTPUT_MODE="raw_json" \
bash scripts/coco_start.sh --no-reinstall -- -p "$(cat runs/task_001/prompt.txt)"
```

更推荐服务端直接传 prompt 参数而不是 shell 拼接，避免转义问题。上面的命令只用于说明环境变量和入口关系。

### 14. 并发与隔离

MVP 可以单 worker 顺序执行。进入生产化后再做并发。

```text
+----------------------------------------------+
| Task Queue                                   |
+----------------------------------------------+
       |                  |                  |
       v                  v                  v
+----------------+ +----------------+ +----------------+
| worker-1       | | worker-2       | | worker-3       |
| workspace A    | | workspace B    | | workspace C    |
| plugin name A  | | plugin name B  | | plugin name C  |
+----------------+ +----------------+ +----------------+
```

并发隔离要求：

| 隔离项 | 要求 |
| --- | --- |
| `COCO_PLUGIN_NAME` | 每个 worker/task 唯一，避免插件名冲突 |
| `COCO_SKILL_DIR` | 每任务独立 bundle |
| workspace | 写代码任务必须独立 worktree |
| `coco_result/` | 最好重定向到任务目录，避免共享状态冲突 |
| secrets | 每个任务只拿到必要凭据 |
| timeout | 每任务强制超时和取消 |

当前 `coco_start.sh` 固定使用仓库根下 `coco_result/`，生产化时建议增加环境变量：

```text
COCO_RESULT_DIR=<runs/task_id/coco_result>
```

这样多个服务端 worker 不会抢同一个 `coco_result/run.log`。

### 15. 监听 Agent 的任务类型设计

建议先支持三类任务，不要一开始开放所有能力：

| 类型 | profile | 是否写代码 | 是否允许 git 写入 |
| --- | --- | --- | --- |
| 分析任务 | `analysis` | 否 | 否 |
| 开发任务 | `develop` | 是 | 否 |
| 交付任务 | `delivery` | 是 | 是，需审批 |

任务类型路由：

```text
+----------------------------------------------+
| prompt intent                                |
+----------------------------------------------+
       |                  |                  |
       v                  v                  v
+----------------+ +----------------+ +----------------+
| 分析/总结/设计 | | 修改代码/补测试 | | 提交/推送/MR  |
+----------------+ +----------------+ +----------------+
       |                  |                  |
       v                  v                  v
+----------------+ +----------------+ +----------------+
| analysis       | | develop        | | delivery       |
| profile        | | profile        | | + approval     |
+----------------+ +----------------+ +----------------+
```

### 16. 可观测性与状态机

服务端状态机：

```text
+----------------------------------------------+

+----------------------------------------------+

+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
```

### 17. 安全边界

服务端监听 Agent 必须默认保守：

| 风险 | 保护措施 |
| --- | --- |
| 任意 prompt 触发危险技能 | profile + risk level + approval |
| 外部系统凭据泄露 | secrets 只通过环境注入，禁止写入仓库 |
| 多任务共享 workspace 污染 | 每写任务独立 worktree |
| 插件名冲突 | 每任务唯一 `COCO_PLUGIN_NAME` |
| 技能被未审查更新 | registry + tests + reload gate |
| 长任务卡死 | timeout + heartbeat + cancel |
| 伪造成功 | exit code + run.log + evidence gate |

每个阶段的验收：

## 可以复用的思维框架

### 框架一：协议 -> 技能 -> 证据 -> 回归

适用场景：任何希望让 Agent 稳定执行复杂工程任务的系统。

```text
+----------------------------------------------+
| Protocol                                     |
| 定义边界、顺序、禁止事项                    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| Skill                                        |
| 把重复动作封装成稳定入口                    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| Evidence                                     |
| 每个关键动作产生日志、manifest、测试结果    |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| Regression                                   |
| 把成功/失败任务转成可重复评测 case          |
+----------------------------------------------+
```

使用方法：

| 步骤 | 问题 |
| --- | --- |
| 协议 | 哪些行为必须做？哪些行为绝对禁止？ |
| 技能 | 哪些动作不该让 Agent 每次自由发挥？ |
| 证据 | 如何证明它真的做了，而不是说做了？ |
| 回归 | 这个任务以后如何自动检查不退化？ |

### 框架二：SSoT 0/1 验收框架

适用场景：需求复杂、容易被语言模糊化的任务。

```text
+----------------------------------------------+
| 需求描述                                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 拆成可观察行为                              |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 为每个行为定义 0/1 判定                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 绑定测试、日志、指标或命令                  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 执行后只看证据，不看主观解释                |
+----------------------------------------------+
```

模板：

| 字段 | 说明 |
| --- | --- |
| Requirement | 要实现什么 |
| Success Criteria | 如何判定成功 |
| Evidence | 哪个命令、日志、测试、指标证明成功 |
| Failure Mode | 什么情况算失败 |
| Owner | 谁负责设计、实现、验收 |

### 框架三：角色权力分离框架

适用场景：一个 Agent 容易边做边改标准、自我验收、跳过验证。

```text
+----------------+  +----------------+  +----------------+
| Commander      |  | Designer       |  | Coder          |
| 是否可做/拆解  |  | 方案/验收标准  |  | 实现/验证      |
+----------------+  +----------------+  +----------------+
         |                   |                   |
         v                   v                   v
+----------------+  +----------------+
| Evaluator      |  | Evolver        |
| 是否接受结果   |  | 失败复盘/下轮  |
+----------------+  +----------------+
```

迁移原则：

| 原则 | 含义 |
| --- | --- |
| 决策和执行分离 | 执行者不应该单独修改验收标准 |
| 实现和评价分离 | 写代码的人不应该只靠自评通过 |
| 失败和演进分离 | 失败不是结束，要转成下一轮约束 |
| 技能按角色隔离 | 不同角色只获得必要能力 |

### 框架四：真实执行优先框架

适用场景：构建、测试、性能、部署、数据链路等不能靠 mock 判断的任务。

核心判断：

| 如果任务涉及 | 则必须有 |
| --- | --- |
| 构建 | 真实 build 命令和日志 |
| 测试 | 真实 test 命令、测试数量、退出码 |
| 性能 | baseline、实验值、统计口径 |
| 稳定性 | 复现路径、压力条件、资源观察 |
| 运维判断 | 指标、日志、查询或看板入口 |

这个框架的底线是：没有真实证据，只能说“无法核对”，不能说“已验证”。

### 框架五：能力目录框架

适用场景：团队规则很多，单靠文档容易遗漏。

模板：

```json
{
  "id": "CAP-XXX",
  "name": "能力名称",
  "trigger_paths": ["哪些文件变化会影响该能力"],
  "required_artifacts": ["必须存在的文件或产物"],
  "tests": ["覆盖该能力的测试"],
  "jenkins_jobs": ["持续验证该能力的任务"],
  "pass_fail": "通过/失败判定"
}
```

价值：

| 传统规则文档 | 能力目录 |
| --- | --- |
| 人读，容易忘 | 机器可读，可检查 |
| 变更影响不清楚 | trigger path 可反查影响 |
| 测试和规则脱节 | 每条能力绑定测试 |
| Jenkins 任务散落 | 每条能力绑定 job |

### 框架六：质量评估分层框架

适用场景：需要长期评估 Agent 输出质量，而不是只看 pass/fail。

```text
+----------------------------------------------+
| 底线回归 expect                              |
| 判断是否过线                                |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 规则质量分 quality                           |
| 判断哪些维度强、哪些维度弱                  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| AI judge                                     |
| 补足关键词规则的机械性                      |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 趋势报告                                     |
| 判断能力是否提升或退化                      |
+----------------------------------------------+
```

关键取舍：

| 做法 | 原因 |
| --- | --- |
| 先有底线，再有评分 | 分数不能替代红线 |
| 规则分为主，AI judge 为辅 | 保留稳定性和可解释性 |
| 维度名稳定 | 便于长期趋势对比 |
| 真实任务沉淀 case | 回归集来自实际失败和交付 |

### 框架七：经验外部化框架

适用场景：希望 Agent 越做越懂项目，而不是每次重新开始。

```text
+----------------------------------------------+
| 任务过程                                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 设计理解 -> design                           |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 踩坑根因 -> pitfalls                         |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 稳定约束 -> constraints                      |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 真实案例 -> regression case                  |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 索引检索 -> experience index                 |
+----------------------------------------------+
```

沉淀原则：

| 知识类型 | 推荐位置 |
| --- | --- |
| 通用稳定约束 | `memory/project/<project>/constraints/` |
| 项目索引 | `memory/project/<project>/README.md` |
| 模型相关上下文 | `memory/project/<project>/<model>/agent_context.md` |
| 可复用学习文档 | `recallfs/projects/<project>/docs/` |
| 回归样本 | `ai/agent_regression/cases/` |

## 这个项目的设计哲学

### 1. Agent 不可信，但过程可以变可信

这个仓库默认不相信 Agent 的自述，而是要求它留下证据：

| 自述 | 可信替代 |
| --- | --- |
| 我已经测试 | 测试命令、退出码、测试数量、日志 |
| 我已经构建 | 构建脚本、构建日志、产物路径 |
| 我理解需求 | SSoT、验收标准、风险清单 |
| 我修复了问题 | 复现前失败、修复后通过 |
| 我沉淀了经验 | memory/pitfall/case 文件路径 |

### 2. 自由生成要被工程结构包住

Agent 的优势是理解和生成，但工程交付需要：

| Agent 自由能力 | 工程约束 |
| --- | --- |
| 生成方案 | 必须落到 design |
| 生成代码 | 必须通过 build/test |
| 生成判断 | 必须被 evaluator/gate 复核 |
| 生成经验 | 必须写入 memory/recallfs/case |

### 3. 失败是资产，不是噪音

项目通过 pitfall、quality case、evolution task 把失败变成下一次的约束。

```text
+----------------------------------------------+
| 失败                                         |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 定位根因                                     |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 沉淀 pitfall / regression case              |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 进入 daily / weekly / quality_captured       |
+----------------------------------------------+
                       |
                       v
+----------------------------------------------+
| 后续自动防回归                               |
+----------------------------------------------+
```

这也是最值得复用的地方：不要只修一次 bug，要让这个 bug 以后更难重现。

## 可迁移到其他项目的方法

### 最小迁移版本

如果要把这套思想迁移到另一个研发项目，最小集合是：

| 模块 | 最小实现 |
| --- | --- |
| 协议 | 一份 `AGENTS.md` 或项目开发协议 |
| 技能 | `clone/build/test/commit` 等稳定脚本 |
| 项目入口 | `env/<project>/scripts/build.sh`, `test.sh` |
| 证据 | 每次运行保存 summary、log、exit code |
| 回归 | 把真实失败任务写成 case |
| 经验 | 把设计和坑点写入可检索目录 |

### 推荐落地顺序

```text
+-----------------------------+
| 1. 收敛 build/test 命令      |
+-----------------------------+
              |
              v
+-----------------------------+
| 2. 写协议和禁止事项          |
+-----------------------------+
              |
              v
+-----------------------------+
| 3. 封装技能入口              |
+-----------------------------+
              |
              v
+-----------------------------+
| 4. 要求证据 manifest         |
+-----------------------------+
              |
              v
+-----------------------------+
| 5. 建立 commit/MR gate       |
+-----------------------------+
              |
              v
+-----------------------------+
| 6. 真实任务沉淀 regression case |
+-----------------------------+
              |
              v
+-----------------------------+
| 7. 多 Agent 角色拆分和自动演进 |
+-----------------------------+
```

不要一开始就追求完整多 Agent。最先要解决的是：能不能稳定、真实、可复核地完成一次工程动作。

## 风险与待改进点

| 风险 | 说明 |
| --- | --- |
| 协议仍可能强于机器门禁 | 文档要求多，部分流程仍依赖 Agent 自觉 |
| 多 Agent 自评可能偏乐观 | evaluator 如果不强制读取真实 artifact，仍可能被自述污染 |
| 技能链接存在同步成本 | `skill/`, `.agents/skills/`, `.trae/skills/` 需要保持一致 |
| 回归 case 质量决定长期价值 | 如果 case 只测口号，不测证据，会稀释信号 |
| 真实项目验证成本高 | clone/fetch/build/test 可能慢，但这是可信度的代价 |

改进方向应继续围绕一个原则：把人为协议转成机器可检查证据。
